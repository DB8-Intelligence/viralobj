## 🔑 Decisões de Arquitetura

### White-label por CSS Custom Properties
O theming por nicho é feito via CSS vars no `globals.css` + atributo `data-niche` no `<html>`:
```css
[data-niche="beleza"] { --brand-500: #ec4899; }
[data-niche="tecnico"] { --brand-500: #f97316; }
/* etc. */
```
O `DashboardShell` aplica isso via `useEffect` quando o nicho do tenant muda.

### Multi-tenant com RLS
Toda query de dados usa `get_tenant_id()` nas políticas RLS do Supabase.
O frontend nunca precisa filtrar por `tenant_id` manualmente — o Supabase resolve.

### Onboarding flow
```
/cadastro
  Step 1: conta (email + senha) → não cria no Supabase ainda
  Step 2: nicho (grid visual 10 opções)
  Step 3: dados do negócio (nome + WhatsApp)
  Step 4: plano → AQUI chama supabase.auth.signUp() + setup_tenant() RPC
```

### Rate limiting (API IA)
Implementado com `Map` em memória (suficiente para MVP). Para produção, usar Redis/Upstash.
Limite: 10 req/min por tenant.

### Plan gating
- `isPlanAtLeast('pro_plus')` — verifica hierarquia: trial < starter < pro < pro_plus < enterprise
- Agente IA Contador: planos `pro_plus` e `enterprise`
- Geração de conteúdo social: planos `pro`, `pro_plus`, `enterprise`
